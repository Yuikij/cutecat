# CuteCat 重构设计草案

## 当前判断

现在的 CuteCat 更像是“聊天 App + 一只会回应的小猫”。用户说一句、点一下，小猫用本地模型生成一句反馈。这条路的问题是：当前本地小模型不适合承担深度对话，长聊容易变成重复、空泛、叽叽喳喳。

所以重构方向不是继续强化聊天，而是降低聊天权重，把模型从“主角”改成“调味料”。

新的核心判断：

> CuteCat 不应该主要是一只会聊天的猫，而应该是一只会自己过日子的猫。

## 设计目标

- 少聊天，多观察。
- 少回复，多行为。
- 少问答，多养成意外。
- 让乐趣来自状态、行为、记忆、习惯和意外，而不是每一句模型回复有多聪明。
- 让本地小模型只负责短文本风味：吐槽、日记、商品描述、事件包装、心情短句。

## 旧循环的问题

当前体验大致是：

```text
用户行为 -> 小猫状态变化 -> LLM 生成一句话 -> UI 展示
```

这会让用户持续期待“小猫说得有趣”，而小模型很难稳定满足这个期待。

更适合电子宠物的循环应该是：

```text
时间流逝 -> 小猫自己做事 -> 用户观察异常 -> 用户干预或放任
-> 产生后果 -> 小猫形成习惯、偏好或记忆 -> 下次行为变化
```

也就是说，小猫的重点不是“回答”，而是“活着”。

## 新核心体验

用户打开 App 时，看到的不是一只静态等待输入的猫，而是一只正在做事的猫。

例子：

- 它把小鱼干藏到被窝里。
- 它因为昨天被摸肚子，今天背对着主人。
- 它给自己起了一个奇怪称号。
- 它半夜写了一条猫日记。
- 它把洗澡视为重大背叛。
- 它捡回来一个没用但有故事的破烂。
- 它钻进纸箱里，不愿意出来。
- 它饿了以后开始啃界面上的按钮。

这类体验对模型要求低，但对玩家更有观察和干预的欲望。

## 玩法结构

### 1. 身体状态层

保留饥饿、精力、清洁、健康、心情等基础状态，但不要只以数值呈现。

状态应该直接驱动可见行为：

- 饿了：翻东西、啃 UI、盯着食物、对商店更敏感。
- 困了：行动变慢、躲起来、拒绝互动。
- 脏了：自己舔毛、在角落蹭来蹭去、引发健康风险。
- 不开心：背对用户、拒绝摸头、把物品藏起来。
- 健康差：活动减少、事件变少、需要照顾。

状态变化应该优先改变猫的行为，而不是只触发一句话。

### 2. 习惯和性格层

小猫的性格不应该只是预设，而应该根据玩家长期行为慢慢长歪。

可能的演化方向：

- 经常摸头：变粘人，更常主动靠近。
- 经常摸肚子：根据好感可能变信任，也可能记仇。
- 经常训它：变阴阳怪气、警惕、叛逆。
- 经常不理它：变独立，开始自娱自乐。
- 经常买怪东西：变收藏癖，主动捡破烂。
- 经常喂食：变贪吃，对食物事件更敏感。
- 经常语音聊天：变话痨，但仍只短句表达。

性格应该影响：

- 主动行为池。
- 随机事件倾向。
- 商店偏好。
- 对同一互动的反应。
- 猫日记内容。
- UI 中的待机动画和小动作。

### 3. 事件记忆层

不追求记住所有聊天内容，而是记住少量重要事件。

适合长期记忆的事件：

- 第一次见面。
- 第一次复活。
- 第一次被摸肚子后咬人。
- 连续多天没被喂。
- 最喜欢的食物。
- 最讨厌的互动。
- 某个宝物的来历。
- 主人某天情绪很差。
- 某次随机事件造成了明显后果。

这些记忆应该能反过来影响后续行为，而不是只作为聊天上下文。

例子：

- 记住“主人总摸肚子”：猫可能提前护住肚子。
- 记住“某个罐头吃坏肚子”：下次看到类似商品会抗拒。
- 记住“主人三天没来”：猫变得冷淡或过分粘人。
- 记住“复活过一次”：解锁特殊称号或梦境事件。

## 聊天的新定位

聊天应该从主玩法降级为辅助玩法。

它不是深度对话入口，而是“偶尔逗猫”的入口。

小猫可以：

- 只抓关键词回应。
- 听不懂长句，但记住用户情绪。
- 给出离谱建议。
- 直接睡着。
- 把用户的话收藏成“人类怪话”。
- 用贴纸式短句、心情词、表情回应。

这样小模型的弱点可以转化成角色设定：不是模型笨，而是猫本来就不认真听人讲话。

## 商店的新定位

商店不应只是“LLM 生成 JSON 商品”的功能展示，而应该成为世界变化入口。

商店卖的东西应该影响猫的行为，而不只是进背包。

商品例子：

- 纸箱：猫更常钻进去，社交欲下降。
- 发光鱼干：今晚猫会发光，第二天精力下降。
- 反重力毛线球：随机事件概率上升。
- 旧铃铛：猫主动找用户的频率上升。
- 可疑罐头：可能恢复健康，也可能闹肚子。
- 主人味道贴纸：粘人倾向上升。
- 空白日记本：增加猫日记出现概率。
- 破旧王冠：触发中二或自恋行为。

LLM 可以继续负责商品名字和描述，但商品效果最好由规则系统控制。

## 猫日记

猫日记可以成为新的核心记忆表现。

每天或每几个关键事件后，小猫写一条很短的日记：

```text
今天主人又摸我肚子。可疑。必须防备。
```

```text
我捡到一个瓶盖。它很圆，可能是月亮掉下来的。
```

```text
主人今天说话很累。我没有听懂，但我趴近了一点。
```

日记的作用：

- 作为轻量长期记忆。
- 给玩家提供“它真的在生活”的感觉。
- 让小模型生成短文本，降低难度。
- 作为之后事件、行为、称号的素材。

## 主界面改造方向

主界面应该从“状态面板 + 操作按钮 + 聊天泡泡”转向“观察一个小生物”。

重点展示：

- 猫当前正在做什么。
- 猫所在的小空间。
- 少量可交互物品。
- 当前状态用行为表达，而不是强依赖数值。
- 聊天入口弱化。
- 操作按钮可以更情境化，而不是永远固定一排。

可能的主界面结构：

```text
顶部：名字、少量状态标记、设置
中间：猫的小房间 / 当前行为 / 可交互物品
底部：根据当前场景变化的几个操作
隐藏层：背包、日记、记忆、称号、商店
```

## 行为系统草案

给猫建立一个“当前行为”概念。

行为可以有：

- id
- 标题
- 动画类型
- 持续时间
- 触发条件
- 可被用户如何打断
- 结束后造成什么影响
- 是否写入记忆
- 是否需要 LLM 生成一句短描述

示例：

```text
行为：藏食物
触发：饥饿低、背包有食物、贪吃性格高
表现：猫把食物拖到角落
用户选择：阻止 / 假装没看见 / 帮它藏
后果：影响好感、信任、贪吃倾向
记忆：主人是否尊重了它的小秘密
```

## 模型使用原则

本地小模型应该做：

- 生成 1 句猫日记。
- 生成 1 句吐槽。
- 给规则事件包装文案。
- 给商店商品起名和描述。
- 根据记忆生成短心情。

本地小模型不应该承担：

- 长篇深度对话。
- 复杂推理。
- 稳定 JSON 业务逻辑。
- 核心数值平衡。
- 游戏规则裁决。

规则系统负责骨架，模型负责味道。

## 重构路线

### 第一阶段：设计收敛

- 明确 CuteCat 的核心循环。
- 确定哪些玩法保留，哪些降级。
- 设计行为系统的数据结构。
- 设计猫日记的数据结构。
- 设计记忆如何影响行为，而不是只影响聊天。

### 第二阶段：代码拆分

- 从 `PetStore` 拆出行为系统。
- 从 `PetStore` 拆出商店系统。
- 从 `PetStore` 拆出事件系统。
- 从 `PetStore` 拆出成就/称号系统。
- 把 prompt 统一收束到独立模块。

### 第三阶段：主界面改造

- 主界面优先展示猫当前行为。
- 弱化聊天泡泡。
- 操作按钮根据场景变化。
- 增加猫日记入口。
- 增加背包物品对场景的影响。

### 第四阶段：记忆与习惯

- 重要事件写入长期记忆。
- 记忆影响行为选择。
- 长期互动形成性格倾向。
- 日记成为可读、可回溯的记忆层。

## 第一版落地记录

已经先实现一个小切片，目标是让 CuteCat 从“点按钮换一句话”开始转向“打开后看到一段羁绊关系”。

已落地：

- 新增猫日记数据 `CatDiaryEntry`，每天打开会生成一条日记。
- 新增羁绊猫格描述：根据好感、观察、陪伴、性格倾向生成“我的猫是什么样”。
- 主界面新增猫日记入口。
- 主操作移除商店入口，弱化功能堆叠。
- 主操作移除“管教”，换成“观察”和“陪伴”。
- 用户聊天中出现疲惫、孤单、难过、开心等情绪时，会写入记忆和猫日记。
- 档案页新增羁绊卡、猫格分享文案、猫日记预览。
- 原“购物狂”称号改为“日记收藏家”，避免商店被砍后出现不可达成称号。

第一版还没解决：

- 猫当前行为还只是文案层，没有独立行为状态机。
- 16人格/分享卡还没有做成正式页面。
- ASCII 动画还没有跟行为系统深度绑定。
- 商店代码仍保留在项目里，只是从主体验移除。
- 还需要在真机/Xcode 环境验证编译和体验节奏。

## 第二版调整记录

针对讨论补了两个关键方向：

### 情绪识别交给小模型

用户输入里的情绪不再只靠关键词判断。

现在流程是：

```text
用户输入 -> LLM 做情绪分类 -> 输出固定标签 -> 规则系统决定猫怎么反应
```

标签包括：

- tired
- lonely
- sad
- anxious
- angry
- happy
- none

这样小模型只做它比较擅长的“短文本分类”，不负责深度安慰，也不直接决定关系结果。

如果模型失败，再退回关键词兜底。

### 猫不一味讨好主人

猫的回应开始受好感度和性格影响。

观察：

- 高好感时，猫会接受被看见。
- 低好感或防备性格时，猫可能只留一个耳朵、躲远一点。
- 观察通常比直接亲近更安全，更适合建立初期信任。

陪伴：

- 高好感时，猫会贴近、被安抚。
- 低好感时，猫可能拒绝安慰，甚至轻微掉好感。
- 傲娇、厌世性格需要更高好感才会柔软。

听见用户情绪：

- 猫会记住用户状态，但不一定马上安慰。
- 好感低时，它可能只是听见、保持距离、写进日记。
- 好感高时，它才更像“陪在你身边”。

新的原则：

> 治愈不是猫无条件讨好用户，而是用户慢慢获得一只小生命的信任。

## 第三版调整记录

### 16猫格报告

新增“16猫格测试”方向，服务分享欲和独特性。

猫格不是随机生成，而是从四个轴计算：

- 社交性：独处 vs 主动
- 安全感：防备 vs 信任
- 混乱度：稳定 vs 离谱
- 亲密度：疏离 vs 贴贴

四个轴组合成类似 `ESCA` 的猫格代码，再映射到一个可分享名称。

例子：

```text
ESCA 太阳纸箱暴君
热闹、信任、离谱，还会主动把爱藏进纸箱。
```

```text
IGRA 慢热月光猫
很慢很慢地靠近，一旦信任就很珍贵。
```

这样用户分享的不是“我抽到了某只猫”，而是：

> 我把它养成了这种猫。

已落地：

- 档案页新增 16猫格报告卡。
- 猫日记页顶部展示今日猫格。
- 猫格报告包含四轴数值和可分享文案。

### 当前生活场景

主界面不再长期显示最新猫日记。

现在主界面显示“小猫当前正在过什么生活”，例如：

- 搜寻食物
- 低电量
- 调查世界
- 嘴硬待机
- 深夜放空
- 小日子

猫日记保留为弹出提示和独立页面记录，避免主界面被一条旧日记卡死。

## 第四版调整记录

### 行为状态机

新增 `currentBehavior`，让小猫拥有明确的当前行为，而不是只有心情和一句文案。

行为类型包括：

- 小日子
- 等待/靠近
- 躲起来
- 搜寻食物
- 低电量睡觉
- 舔毛
- 调查世界
- 护住肚子
- 写日记
- 展示/炫耀
- 闹别扭

行为会被写入存档，并影响：

- 主界面的生活场景文案。
- ASCII 动画使用的心情态。
- 事件触发时猫的表现。
- 观察、陪伴、听见用户情绪后的后续状态。

### 规则事件骨架

随机事件不再依赖 LLM 生成 JSON。

现在由规则系统根据猫状态选择事件，例如：

- 饿了触发“碗底宇宙”。
- 困了触发“低电量团子”。
- 低好感/傲娇触发“纸箱边界线”。
- 好奇/中二触发“右爪实验”。
- 有日记后触发“日记被风翻开”。

每个事件的选项和数值后果由代码控制，模型不再负责裁判。

### 分享卡

新增 16猫格分享卡页面，适合截图传播。

分享卡包含：

- 猫名。
- 猫格代码。
- 猫格名称。
- 四轴数值。
- 代表行为。
- 最新猫日记。

### ASCII 绑定行为

ASCII 动画仍然保留，但它现在优先跟随 `currentBehavior` 映射出的心情，而不是只看基础状态。

这让“护住肚子”“写日记”“调查世界”“低电量”等行为能在视觉上更明显。

### 商店路径

商店已经从主界面和核心循环中移除。

当前代码里仍保留旧商店文件和背包字段，原因是避免一次性删除 Xcode 工程引用造成构建风险。后续如果确认不回滚，可以再做一次纯清理提交：

- 删除 `ShopView`。
- 删除 `generateShopItems/buyItem/feedItemToCat` 等旧商店逻辑。
- 清理 `ShopItem` 和 `inventoryItems`。
- 清理 Xcode 工程文件引用。

## 第五版调整记录

### 首屏 10 秒钩子

主界面新增“第一眼卡片”，打开 App 立刻看到：

- 猫格代码。
- 猫格名称。
- 当前生活场景。
- 代表行为。

这张卡点击后进入档案页，目标是让用户在首屏就理解：

> 这不是普通宠物状态面板，而是我养出来的一只独特小猫。

### 关系变化反馈

所有好感度变化统一通过 `adjustAffinity` 出口显示反馈。

现在互动后会出现类似：

```text
💕 小猫更靠近了一点  好感+2
```

或：

```text
🫥 小猫更防备了一点  好感-1
```

这样用户能更清楚地感到：自己的行为真的改变了关系。

### 分享卡视觉强化

16猫格分享卡改成更像测试结果图：

- 顶部彩色身份区。
- 大号猫格代码。
- 猫格名称和解释。
- 四轴数值。
- 代表行为卡片。
- 最新猫日记卡片。
- 底部传播语：“我养出来的，不是抽到的”。

## 第六版调整记录

### 极端猫格名场景

为了让性格更分明，新增一组可触发的“名场景”。

它们不是纯随机段子，而是由关系状态和长期行为触发：

- **冷藏级喜欢**：好感极高、陪伴很多时，小猫会因为太喜欢主人，研究“喜欢的东西应该放进冰箱保存”。
- **梦里复仇**：好感很低、被管教多时，小猫会梦见自己把主人揍赢。
- **离家出走预演**：心情很低、长期无聊时，小猫会把小包袱拖到门口，练习离家出走。

这些场景的目的：

- 让小猫不只是“可爱”或“治愈”，而是有记仇、占有欲、边界感和荒诞脑回路。
- 让玩家能截图分享“我家猫今天疯成这样”。
- 让极端养成路径产生强记忆点。

## 待讨论问题

- 猫应该更像电子宠物、桌宠，还是养成游戏？
- 主界面是否需要一个固定“小房间”？
- 用户每天打开 App 的第一件趣事应该是什么？
- 猫日记是每天一条，还是由事件触发？
- 聊天入口应该隐藏到二级页面，还是保留在底部？
- 商店商品效果应该固定模板，还是允许模型部分决定？
- 小猫是否应该有“主动打扰用户”的能力？
- 死亡和复活机制是否保留？如果保留，应该更轻还是更重？

## 第二轮讨论：玩法方向收敛

### 1. 商店是否保留

当前判断：商店可以砍掉，或者至少从主玩法里移除。

原因：

- 商店现在更像一个“LLM 生成 JSON”的功能展示，不是核心乐趣。
- 对养成关系没有强绑定，买东西和猫变得独特之间的联系不够强。
- 它会分散设计焦点，让项目从“养一只独特的小猫”变成堆功能。

如果未来还想保留“获得物品”的体验，可以把商店降级成其他系统的一部分：

- 随机事件中获得物品。
- 猫自己捡回物品。
- 每日抽取小物件。
- 根据猫性格解锁专属收藏。

也就是说，可以保留“物品”和“收藏”，但不一定保留“商店”。

### 2. 抽卡感和养成感

这两个方向值得深入。

抽卡感提供短期刺激：

- 今天猫会长出什么倾向？
- 今天会触发什么事件？
- 今天会写什么日记？
- 今天会捡到什么东西？
- 这次互动会不会解锁新反应？

养成感提供长期牵引：

- 用户长期行为改变猫的性格。
- 小猫形成独特偏好、习惯、阴影和依赖。
- 同样的输入，在不同用户的小猫身上出现不同结果。
- 猫不是被抽到的，而是被养出来的。

理想关系：

```text
抽卡感 = 每次打开都有未知惊喜
养成感 = 这些惊喜越来越像“我这只猫才会发生的事”
```

抽卡不一定是传统卡池，也可以是：

- 性格碎片。
- 行为变体。
- 猫日记。
- 梦境。
- 称号。
- 纪念物。
- 稀有动作。
- “主人观察报告”。

### 3. 独一无二的小猫

这是最应该深入的核心卖点。

如果 CuteCat 想变得容易传播，重点不是“这只猫很聪明”，而是：

> 这是我养出来的猫，和你的不一样。

独特性可以由几层组成：

- 初始随机：出生时带有一点不同天性。
- 行为养成：用户怎么对它，它就怎么长歪。
- 记忆沉淀：少量关键事件成为它的私人经历。
- 表达变体：同一性格有不同口癖、动作、日记风格。
- 稀有转变：长期行为触发罕见人格或称号。

需要避免的问题：

- 只是随机标签，不是真独特。
- 只是数值不同，玩家感觉不到。
- 性格只影响文案，不影响行为。

真正的独特应该能被用户观察到：

- 它是不是主动靠近。
- 它是不是记仇。
- 它是不是爱藏东西。
- 它是不是话多。
- 它是不是容易 emo。
- 它是不是喜欢某个物品。
- 它是不是害怕某个互动。

### 4. 输入与输出链路

当前链路：

```text
用户输入文字/语音/按钮 -> 小猫生成回应 -> 动作 + TTS
```

问题是输出太依赖小模型说得好不好。

新链路应该变成：

```text
用户输入 -> 系统理解为一个意图或事件 -> 改变猫状态/记忆/行为
-> 选择猫的动作/反应 -> 小模型只补一句短风味文本
-> 输出动画、TTS、日记、事件结果或长期变化
```

输出侧的原则：

- 先有行为，再有语言。
- 先由规则决定结果，再让模型包装。
- TTS 不一定每次都说完整回复，可以只说短句、拟声、抱怨、拒绝。
- 模型输出可以藏在日记、梦话、观察报告、事件描述里，降低对即时对话质量的依赖。

输入侧除了固定按钮，还可以有：

- 自由摸猫：点击/长按/拖动不同身体区域。
- 投放物品：把获得的东西拖给猫或放进房间。
- 观察/不打扰：什么都不做也是输入。
- 选择事件分支：随机事件给 2-3 个反应方式。
- 情绪投喂：用户告诉猫“我今天很累/开心/烦”，系统记录情绪，不要求深聊。
- 命名和贴标签：给物品、日记、猫行为起外号。
- 训练/习惯暗示：鼓励或制止某个行为。
- 每日签名动作：用户每天用一个小动作打招呼。
- 语音语气：不只识别文字，也可把“用户说话频率/是否长时间沉默”作为轻量信号。

### 5. 分享欲强的玩法

分享点应该围绕“我的猫是什么样”展开。

可分享内容：

- 小猫 16 人格测试。
- 猫格雷达图。
- 今日猫日记截图。
- 本周养成报告。
- 猫对主人的评价。
- 稀有称号。
- 猫的怪癖清单。
- 我和猫的关系诊断。
- 猫的 MBTI 式标签，但不要完全照搬人类 MBTI。

例子：

```text
你的猫是：ENFP 型纸箱暴君
外向程度：只对饭外向
安全感：低
记仇程度：极高
代表行为：把你的关心拖进纸箱里藏起来
```

这类内容天然适合截图传播，因为用户分享的不是功能，而是“我养出来了一个离谱人格”。

### 6. ASCII 动画是否保留

ASCII 动画可以保留，但不一定要作为唯一表现形式。

它的优点：

- 有辨识度。
- 成本低。
- 适合表达抽象和搞怪。
- 可以弱化画面资产压力。
- 和“小模型、小本地、奇怪电子宠物”的气质相配。

它的问题：

- 长期表现力有限。
- 新用户第一眼可能不够直观。
- 分享截图可能需要更强视觉识别。

当前建议：

- 保留 ASCII 作为核心风格资产。
- 强化它的“行为动画”，不要只是表情变化。
- 搭配房间、物品、日记卡、人格报告等 UI，让整体更完整。
- 后续可以加入少量像素风/贴纸风视觉元素，但不要急着推翻 ASCII。

结论：

> ASCII 可以继续做灵魂，但需要一个更完整的舞台。
